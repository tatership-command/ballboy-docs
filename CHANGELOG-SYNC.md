# Changelog sync transform

This documents the exact, deterministic transform from the `ballboy` repo's
release notes (`.docs/releases/v*.md`) into this repo's Release Notes section
(`content/docs/changelog/`), so an automated sync workflow can reproduce it
identically for every future release. The backfill of v1.0.0–v1.0.24 was
generated by this transform (script used: a one-off Python script, not
committed — reimplement per the rules below).

**`scripts/release_to_changelog.py` is the canonical, committed
implementation of this transform** — it is byte-identical against every
existing page in `content/docs/changelog/` (verified at build time by the
`sync-release.yml` workflow's use of it; see `SYNC-SETUP.md` for the
end-to-end automated flow and the manual fallback). Any future change to the
rules below must be applied to that script, not reimplemented ad hoc.

## Source

`ballboy/.docs/releases/v{VERSION}.md`, one file per shipped release. Frontmatter:

```yaml
---
version: v1.0.24
date: 2026-07-08
title: "Team Owners Get League Access"
color: 0xE67E22
---
```

Body: markdown, `## What's New` and/or `## Fixes` sections.

## Output

One page per release at `content/docs/changelog/{version}.md` (e.g.
`v1.0.24.md`), plus the static section index `_index.md` (not regenerated by
sync — hand-authored, edit only if the section intro needs to change).

### Frontmatter mapping

| Output field | Rule |
| --- | --- |
| `title` | `"{version} — {source title}"` (em dash `—`, U+2014). Always double-quoted in the output YAML regardless of source quoting, with `"` and `\` escaped, since source titles can contain `/` (e.g. `/stream`). |
| `summary` | `"{version}: {source title}"` — one-line RAG summary. Same quoting rule as `title`. |
| `date` | Copied verbatim from source `date` (unquoted `YYYY-MM-DD`). |
| `weight` | See formula below. Determines nav order. |
| `color` | **Dropped.** Not carried into the page — it's a Discord-embed-only field; the section `_index.md` explains the green/orange convention in prose instead. |

### Weight formula (newest-first nav order)

hugo-book sorts section children by **ascending** weight, so the newest
release must get the **lowest** weight. Parse the version as `vMAJOR.MINOR.PATCH`
and compute:

```
weight = 100000 - (major * 10000 + minor * 100 + patch)
```

This is monotonically decreasing as the version increases, is reproducible
from the version string alone, and leaves headroom for `major`/`minor` to grow
before colliding with the base offset. If a future version's `patch` (or
`minor`) exceeds two digits (>= 100), widen the multipliers (e.g.
`major*1_000_000 + minor*1_000 + patch`) — keep it consistent across every
page in one migration, don't mix scales.

### Body

Copied through unchanged (`## What's New` / `## Fixes` sections, list items,
bold/code formatting all verbatim). One thing is added: an H1 + dateline
between the frontmatter and the body:

```markdown
# {version} — {source title}

_Released {date}_

{source body, trimmed, trailing newline}
```

## Section landing page (`_index.md`)

Not part of the per-release sync — hand-authored once. Frontmatter:
`title: "Release Notes"`, `weight: 60` (keeps the section last, after the
existing `getting-started(10)/concepts(20)/commands(30)/flows(40)/faq(50)`),
`bookCollapseSection: true` (matches every other top-level section).

## Sync workflow contract

A future automation that watches `ballboy/.docs/releases/v*.md` for new files
should, per new file: read frontmatter + body, apply the table above, write
`content/docs/changelog/{version}.md`, then run the Hugo build gate
(`GOTOOLCHAIN=go1.25.12 hugo --gc --minify`) before committing. It should
never touch `_index.md` or any other existing changelog page.
